# 2.7 VoIP

## 2.7.1 终端通过 H323 无法注册故障排查

终端无法通过 H323 注册问题的排错思路为：先查看配置，再查看网络，然后排查服务器侧能否正常回应请求报文。

### 查看配置

查看 MSR 路由器上的语音相关配置是否正确，查看配置有如下关键点：

- 查看路由器是否配置语音网关使用的源 IP 地址
- 查看路由器是否配置网关别名
- 查看路由是否配置网管对应的主用网守名称和 IP 地址
- 查看路由器是否激活 GK Client 功能
- 查看路由器是否有可注册实体
- 查看路由器是否开启注册功能

**命令：** 进入 voice-setup 视图下使用 `display this` 命令

例如：通过命令查看，可以看到主用 GK 的名字为 gk-server，地址为 172.31.123.13，端口为 1719；备用 GK 的名字为 gk-server-slave，地址为 172.31.123.14，端口为 1719；本端注册使用的源地址为 1.1.1.9；本端的网关名称为 h3c；开启了注册功能。

```bash
[H3C]voice-setup
[H3C-voice]display this
voice setup
gk-client gk-id gk-server gk-addr 172.31.123.13 1719
 //要和 GK 上的 gk-name 一致
gnd-id gk-server-slave gk-addr 172.31.123.14 1719
gw-address 1.1.1.9
 //一般使用路由的 loopback 接口
gw-id h3c
 //要和 GK 上配置的 gw-id 一致
reg on
 //开启注册功能
```

查看路由器是否有可注册实体，可注册实体要求如下：

- 必须为 POTS 实体
- POTS 实体下必须要配置 match-template
- 实体下没有配置 undo register-number 命令

**命令：** 进入 dial-program 视图，使用 `display this` 命令

例如：通过命令可以看到存在可注册实体 1000

```bash
[H3C]voice-setup
[H3C-voice]dial-program
[H3C-voice-dial]display this
//进入 voice setup 视图
//进入 dial-program 视图
dial-program
entity 1000 pots match-template 1000 line 1/0
```

### 排查网络

查看终端设备和 H323 服务器之间是否可达，包括以下两方面：

- 终端和 H323 服务器之间路由可达
- 终端和 H323 服务器之间 1719（注册端口）和 1720（建立会话端口）端口可达

查看终端和路由器之间是否路由可达。

**命令：** `ping x.x.x.x`

例如：通过命令查看，可以确认终端和 H323 服务器之间路由可达。

```bash
[H3C]ping 172.31.123.13
PING 172.31.123.13: 56 data bytes, press CTRL_C to break
Reply from 172.31.123.13: bytes=56 Sequence=1 ttl=255 time=11 ms
Reply from 172.31.123.13: bytes=56 Sequence=2 ttl=255 time=11 ms
Reply from 172.31.123.13: bytes=56 Sequence=3 ttl=255 time=11 ms
Reply from 172.31.123.13: bytes=56 Sequence=4 ttl=255 time=11 ms
Reply from 172.31.123.13: bytes=56 Sequence=5 ttl=255 time=11 ms

--- 172.31.123.13 ping statistics ---
5 packet(s) transmitted
5 packet(s) received
0.00% packet loss
round-trip min/avg/max = 11/11/11 ms
```

路由可达后需要再查看路由器和 H323 服务器之间 1719 端口是否可达。

需要服务器侧配合查看是否能够收到路由器发出的请求报文，如果服务器侧无法收到，请排查网络中是否存在防火墙，如果存在防火墙，请放通相应的协议和端口，如果不存在防火墙，请排查协议报文丢在网络哪个位置。

### 排查服务器是否回复报文

需要协调服务器侧人员查看收到我司发出的 H323 请求报文后，服务器是否能够回应，如果服务器侧没有回应请协调服务器侧处理。

### 服务器回复异常消息

可通过 debugging 命令来查看服务器回复的消息类型，通过服务器回复的类型，可大致判断问题可能性。

**命令：** `debugging voice h323 all terminal debugging terminal monitor`

例如：通过如上命令可以查看到路由器向 GK 网关 172.31.123.13 发送 RRQ 注册报文，然后收到了服务器回复的 RCF，表示注册成功。若 GK 回复 RRJ 消息，则说明服务器拒绝网关注册，确定路由器配置无误后，需要协调服务器侧排查为什么拒绝注册。

```bash
Dec 25 16:13:02:452 2013 H3C IP/VOICE: IPP_RAS [SYS] Send heavy RRQ (4) Msg to GK (172.31.123.13) when receive GCF Msg
//H3VG 向 GK(172.31.123.13) 发送 RRQ 注册报文

Dec 25 16:13:02:452 2013 H3C IPP/7/VOICE: IP_SOCK [SYS] Send RAS Msg On Socket
27 length
75

Dec 25 16:13:02:468 2013 H3C IPP/7/VOICE: IPP_SOCK [SYS] Receive Socket Msg ASYN_READ sockid
27

Dec 25 16:13:02:469 2013 H3C IPP/7/VOICE: IPP_RAS [SYS] Receive RCF Msg, Seq=4, TTL=180
//收到 GK 回复的 RCF 消息，说明 H3VG 已经注册成功
```

## 2.7.2 终端通过 SIP 无法注册故障排查

终端无法通过 SIP 注册问题的排错思路为：先查看配置，再查看网络，然后再排查服务器侧能否正常回应 SIP 请求报文。

### 查看配置

查看 MSR 路由器上的语音相关配置是否正确，查看配置有两个关键点：
1. 查看路由器是否开启 SIP 注册功能
2. 查看路由器是否有可注册实体

查看路由器是否开启注册功能。

**命令：** 进入 SIP 视图下使用 `display this` 命令

例如：通过命令查看，当前主用的 SIP 服务器地址为 172.31.123.121；备用 SIP 服务器地址为 172.31.123.123；当前开了 SIP 注册功能。

```bash
[H3C]voice-setup
[H3C-voice]sip
[H3C-voice-sip]display this
//进入 voice setup 视图
//进入 SIP 视图
sip
registrar ipv4 172.31.123.121
 //主用 SIP 服务器地址
registrar ipv4 172.31.123.123 slave
 //备用 SIP 服务器地址
register-enable on
 //开启注册功能
return
```

查看路由器是否有可注册实体，可注册实体要求：
1. 必须为 POTS 实体
2. POTS 实体下必须配置 match-template
3. 实体下没有配置 undo register-number 命令

**命令：** 进入 dial-program 视图，使用 `display this` 命令

例如：通过命令可以看到存在可注册实体 4000。

```bash
[H3C]voice-setup
//进入 voice setup 视图
[H3C-voice]dial-program
//进入 dial-program 视图
[H3C-voice-dial]display this
dial-program
entity 4000 pots match-template 4000 line 1/0
//需要注册的号码
//此实体对应的接口
return
```

### 排查 VG 与 SIP 服务器是否路由可达

查看终端设备和 SIP 服务器之间是否可达，包括两方面：
1. 终端和 SIP 服务器之间路由可达
2. 终端和 SIP 服务器之间 5060 端口可达

查看终端和路由器之间是否路由可达。

**命令：** `ping x.x.x.x`

例如：通过命令查看，可以确认终端和 SIP 服务器之间路由可达。

```bash
[H3C]ping 172.31.123.121
PING 172.31.123.121: 56 data bytes, press CTRL_C to break
Reply from 172.31.123.121: bytes=56 Sequence=1 ttl=255 time=15 ms
Reply from 172.31.123.121: bytes=56 Sequence=2 ttl=255 time=11 ms
Reply from 172.31.123.121: bytes=56 Sequence=3 ttl=255 time=11 ms
Reply from 172.31.123.121: bytes=56 Sequence=4 ttl=255 time=11 ms
Reply from 172.31.123.121: bytes=56 Sequence=5 ttl=255 time=11 ms

--- 172.31.123.121 ping statistics ---
5 packet(s) transmitted
5 packet(s) received
0.00% packet loss
round-trip min/avg/max = 1/1/1 ms
```

路由可达后需要再查看路由器和 SIP 服务器之间 5060 端口是否可达。

需要服务器侧配合查看是否能够收到路由器发出的请求报文，如果服务器侧无法收到，请排查网络中是否存在防火墙，如果存在防火墙，请放通相应的协议和端口，如果不存在防火墙，请排查协议报文丢在网络哪个位置。

MSR 路由器 SIP 默认基于 UDP，使用默认端口为 UDP 5060，可更改。可选基于 TCP 和 TLS。

**命令：** 进入 SIP 视图下使用 `display this` 命令

例如：通过如上命令可以查看到主用服务器基于 TCP 端口号使用 50000 的 SIP 协议；备用服务器使用默认，基于 UDP，端口号为 5060 的 SIP 协议。

说明：VG 上使用的注册端口取决于 SIP 服务器，若 SIP 服务器没有做特殊设置，那么请使用标准的 5060 端口，若 SIP 服务器要求使用特殊端口，则 VG 侧需要配合 SIP 服务器做更改。

```bash
[H3C]voice-setup
[H3C-voice]sip
[H3C-voice-sip]display this
sip
registrar ipv4 172.31.123.121 port 50000 transport tcp
registrar ipv4 172.31.123.123 port 5060 slave
register-enable on
return
```

### 排查 SIP 服务器能否正常响应 VG 发出的注册报文

需要协调服务器侧人员查看收到我司发出的 SIP 请求报文后，服务器是否能够回应 200 OK 或者 401 鉴权消息，如果服务器侧没有回应请协调服务器侧处理。

说明：
- 200 OK 消息代表服务器同意注册并返回注册成功消息
- 401 消息代表服务器同意注册，但是需要 VG 上送鉴权消息（注册的用户名密码等），VG 重新发送带有鉴权消息的注册报文，如果鉴权通过，则服务器回复 200 OK 消息告知 VG 注册成功
