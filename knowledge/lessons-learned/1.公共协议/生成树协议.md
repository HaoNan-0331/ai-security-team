# 生成树协议

## MSTP故障排查

MSTP定位故障的思路是按照生成树的工作原理：先确定二层网络MSTP的运行范围，再确定全网根桥及域根的一致性，确认域边界，然后再确定每一台参与MSTP计算的网桥的端口角色是否正确，包括在CIST和MSTI中的端口角色是否正确，最后再确定各类生成树保护机制是否导致了异常端口阻塞，业务VLAN是否贯穿全网，是否存在可能的环路。

### 查看生成树状态

大量的交换机默认情况下并不开启生成树协议，因此要确认每一台网桥都开启了生成树功能并且生成树协议模式为MSTP。

**命令：** `display stp`

例如：通过命令查看正确的结果：

```bash
<H3C>display stp
Protocol Status       : enabled
Protocol Std.         : IEEE 802.1s
Version               : 3
```

通过命令查看错误的结果：

```bash
<H3C>display stp
Protocol Status       : disabled
Protocol Std.         : IEEE 802.1s
Version               : 3
```

### 查看并修改生成树基本配置

生成树的基本配置包括四项，所有参与生成树计算的网桥务必保持一致。

**命令：** `display current-configuration`

例如：通过命令查看配置。

- 生成树模式为MSTP。

```bash
[H3C]stp mode mstp
```

MSTP为默认模式，配置不显示。

- 生成树的网络直径。

```bash
[H3C]stp bridge-diameter diameter
```

不建议进行配置，使用默认参数7，配置不显示。

- 生成树的时间参数。

```bash
[H3C]stp timer
```

不建议进行配置，使用默认参数，配置不显示。

### 查看根桥及域根的一致性，确定域边界

查看整个二层网络中相关网桥的总根桥必须为同一个网桥，在同一个域内每个MSTI的域根为同一个网桥。

**命令：** `display stp`

例如：通过命令查看根桥的结果。

```bash
<H3C>display stp
CIST Bridge          : 32768.0023-8914-3ea9
Bridge Times         : Hello 2s MaxAge 20s FwDly 15s MaxHop 20
CIST Root/ERPC       : 0.0023-8939-88b0 / 20
CIST RegRoot/IRPC    : 32768.0023-8914-3ea9 / 0
MSTI Bridge ID       : 4096.0023-8914-3ea9
MSTI Root/IRPC       : 0.0023-8927-adf5 / 20
```

要求所有网桥查询到的根桥必须一致，同一域内的同一MSTI的域根也需要一致，确定域边界。

在域边界通往总根桥的接口状态查询。

**命令：** `display stp interface GigabitEthernet1/0/2`

```bash
<H3C>display stp interface GigabitEthernet1/0/2
----[CIST] [Port1(GigabitEthernet1/0/2)] [FORWARDING]
----[MSTI 1] [Port1(GigabitEthernet1/0/2)] [FORWARDING]
----[MSTI 2] [Port1(GigabitEthernet1/0/2)] [DISCARDING]
 Port Role            : Alternate Port
```

### 查看各网桥互联接口配置

如果出现网桥根桥不一致的情况，需要查看根桥不一致的两个相邻网桥互联端口状态及配置。

- 确认端口STP处于使能状态。

**命令：** `display stp interface interface-type interface-number`

例如：
```bash
<H3C>display stp interface GigabitEthernet1/0/1
----[CIST] [Port1(GigabitEthernet1/0/1)] [FORWARDING]
 Port Protocol         : enabled
```

- 确认端口未处于单通状态。对于域内互联的两个接口，通过显示接口的BPDU收发报文，如果两端口都在发，其中只有一端收到，则说明存在单通现象。

对于域间互联的两个接口，通过显示接口的BPDU收发报文，正常情况下应该是一端发、一端收，且发送BPDU的端口应该有学习到的动态MAC。如果相连的两端口都在发送BPDU，则说明存在单通现象。

```bash
<H3C>display stp interface GigabitEthernet1/0/1
```

- 确认端口上配置未启用和BPDU相关的过滤功能。

**命令：** `display current-configuration interface interface-type interface-number`

```bash
[H3C]display current-configuration interface GigabitEthernet1/0/1
```

- 如果两交换机之间还联有其他二层设备（如IPS、FW、传输设备等），需要确保该二层设备透传BPDU报文。可以在交换机两边对应的端口上启动debug功能，判断收发BPDU报文是否一致。

**命令：** `debug stp packet interface interface-type interface-number verbose`

```bash
<H3C>debug stp packet interface GigabitEthernet1/0/18 verbose
```

- 如果同一个MSTP域被意外分割成多个域，需要查询MSTP region配置。

**命令：** `display current-configuration`

```bash
[H3C]display current-configuration
#
stp region-configuration
 region-name test
 instance 1 vlan 100
 instance 2 vlan 200
 active region-configuration
#
```

注释：同一域内交换机配置region-name必须相同；实例与vlan的映射关系必须相同；修改配置后必须重新执行`active region-configuration`命令，使修改生效。

如若与友商进行MSTP互通，需要在和友商互联接口上配置摘要侦听功能。

**命令：** `stp config-digest-snooping`

### 确认端口角色

确认相关网桥上相关端口角色是否正常。对于MSTP来说，同一个端口在不同的MSTI中角色很可能不同。

重点是排查出现故障的相关vlan（不论是断路还是环路）所经过的路径的相关端口在对应的MSTI中角色是否与预期一致。

**命令：** `display stp interface interface-type interface-number`

例如：如下命令中显示，端口2/0/1在CIST中作为根端口，在MSTI1中作为根端口，而在MSTI2中作为指定端口。

```bash
[H3C]display stp interface GigabitEthernet2/0/1
```

### 查看并修改链路开销标准，确认接口工作模式，确定异常阻塞端口

- 查看网桥上stp链路开销标准的配置是否一致，正确选路的前提是全网标准一致。H3C交换机支持dot1d、dot1t和legacy三种标准。默认是legacy标准，属于H3C公司私有。和友商进行互通时建议使用dot1t标准。

**命令：** `display stp`

例如：通过命令查看，可以确认本网桥选用的stp链路开销标准。

- 修改stp链路开销标准命令：

```bash
[H3C]stp pathcost-standard dot1t
```

- 查看相关网桥互联接口的工作模式（速度和双工模式），确认工作在预设定状态。

### 检查业务VLAN是否贯穿全网（仅指该业务VLAN需要覆盖到的网桥）

查看出现故障的业务VLAN，进行端到端的全部路径的查询。其中，在MSTP域内需要根据对应的MSTI进行路径查询。需要确认以下内容。

- 转发路径上每一个端口（网桥互连端口）上正确配置了相应的业务VLAN。

**命令：** `display interface interface-type interface-number`

```bash
[H3C]display interface GigabitEthernet1/0/1
GigabitEthernet1/0/1 current state: Up
PVID: 1
Mdit type: auto
Port link-type: trunk
VLAN passing: 1(default vlan), 100-110
VLAN permitted: 1(default vlan), 100-110
Trunk port encapsulation: IEEE 802.1q
```

- 转发路径上每一个端口（网桥互连端口）上正确学习到了相关MAC，且多次查看MAC是否频繁迁移。

**命令：** `display mac-address dynamic`

```bash
[H3C]display mac-address 0000-0000-0021
MAC ADDR          VLAN ID  STATE      PORT INDEX
0000-0000-0021    100      LEARNED    GigabitEthernet1/0/2
--- 1 MAC address(es) found ---

[H3C]display mac-address 0000-0000-0021
MAC ADDR          VLAN ID  STATE      PORT INDEX
0000-0000-0021    100      LEARNED    GigabitEthernet1/0/1
--- 1 MAC address(es) found ---
```

如果存在频繁的业务MAC飘移，则很可能存在环路问题。需要进行环路的追踪。追踪的原则是判断业务MAC正常情况下应该从哪个端口收到，那么从其他的端口收到则可能是环路的端口。通常用display看到的端口就是发生环路的端口。

- 查看转发路径上每一个端口（网桥互连端口）上的流量统计情况，重点关注端口收发报文占用带宽的百分比，组播和广播流量大小。

**命令：** `display interface interface-type interface-number`

```bash
[H3C]display interface GigabitEthernet1/0/1
GigabitEthernet1/0/1 current state: Up
Port priority: 0
Peak value of input: 108553678 bytes/sec, at 2000-04-29 07:15:01
Peak value of output: 108553647 bytes/sec, at 2000-04-29 07:15:01
Last 300 seconds input: 822376 packets/sec, 108553662 bytes/sec
Last 300 seconds output: 822375 packets/sec, 108553630 bytes/sec
Input (total): 453736361 packets, 59892409109 bytes
0 unicasts, 453604531 broadcasts, 131834 multicasts
Input (normal): 453736366 packets, 59892409109 bytes
0 unicasts, 453604531 broadcasts, 131834 multicasts
Input: 0 input errors, 0 runts, 0 giants, 0 throttles
0 CRC, 0 frame, 0 overruns, 0 aborts
0 ignored, 0 parity errors
Output(total): 453585222 packets, 59873409459 bytes
0 unicasts, 453579906 broadcasts, 5326 multicasts, 0 pauses
Output (normal): 453585232 packets, 59873409459 bytes
0 unicasts, 453579906 broadcasts, 5326 multicasts, 0 pauses
Output: 0 output errors, 0 underruns, 0 buffer failures
Utilization:  100%  100%
```

从端口统计数据看出，如果端口收发数据的百分比超过90%，并且广播和组播数据占据绝大部分比例时，很可能存在环路。需要追踪此类端口来判断环路路径。判断方式是在环路路径上的端口收发的广播/组播数据量都很大，未在环路路径上的端口仅发方向的广播/组播流量大。

## RSTP故障排查

RSTP定位故障的思路是按照生成树的工作原理：首先确定二层网络RSTP的运行范围，其次确定全网根桥的一致性，再次确定每一台参与RSTP计算的网桥的端口角色是否正确，最后再确定各类生成树保护机制是否导致了异常端口阻塞，业务VLAN是否贯穿全网，是否存在可能的环路。

### 查看生成树状态

确认每一台网桥都开启了生成树功能并且生成树协议模式为RSTP。

**命令：** `display stp`

例如：通过命令查看正确的结果是：

```bash
<H3C>display stp
Protocol Status       : enabled
Protocol Std.         : IEEE 802.1w
Version               : 3
```

通过命令查看错误的结果是：

```bash
<H3C>display stp
Protocol Status       : disabled
Protocol Std.         : IEEE 802.1w
Version               : 3
```

### 查看并修改生成树基本配置

生成树的基本配置包括四项，所有参与生成树计算的网桥务必保持一致。

**命令：** `display current-configuration`

例如：通过命令查看配置。

- 生成树的网络直径。

```bash
[H3C]stp bridge-diameter diameter
```

不建议进行配置，使用默认参数。

- 生成树的时间参数。

```bash
[H3C]stp timer
```

不建议进行配置，使用默认参数。

### 查看根桥的一致性

查看整个二层网络中相关网桥的根桥必须为同一个网桥。

**命令：** `display stp`

例如：通过命令查看根桥的结果。

```bash
<H3C>display stp
CIST Bridge          : 32768.0023-8939-88b0
Bridge Times         : Hello 2s MaxAge 20s FwDly 15s MaxHop 20
CIST Root/ERPC       : 0.0023-8914-3ea9 / 20
CIST RegRoot/IRPC    : 32768.0023-8939-88b0 / 0
```

要求所有网桥查询到的根桥必须一致。

### 查看各网桥互联接口配置

如果出现网桥根桥不一致的情况，需要查看根桥不一致的两个相邻网桥互联端口状态及配置。

- 确认端口STP处于使能状态。

**命令：** `display stp interface interface-type interface-number`

例如：
```bash
<H3C>display stp interface GigabitEthernet1/0/1
```

- 确认端口未处于单通状态。通过显示接口的BPDU收发报文，正常情况下应该是一端发、一端收，且发送BPDU的端口应该有学习到的动态MAC。如果相连的两端口都在发送BPDU，则说明存在单通现象。

**命令：** `display stp interface interface-type interface-number`

```bash
<H3C>display stp interface GigabitEthernet1/0/1
```

- 确认端口上配置未启用和BPDU相关的过滤功能。

**命令：** `display current-configuration interface interface-type interface-number`


- 如果两交换机之间还联有其他二层设备（如IPS、FW、传输设备等），需要确保该二层设备透传BPDU报文。可以在交换机两边对应的端口上启动debug功能，判断收发BPDU报文是否一致。

**命令：** `debug stp packet interface interface-type interface-number verbose`

```bash

```

### 确认端口角色

确认相关网桥上相关端口角色是否正常。重点是排查出现故障的相关VLAN（不论是断路还是环路）所经过的路径的相关端口角色是否与预期一致。

**命令：** `display stp interface interface-type interface-number`

### 检查业务VLAN是否贯穿全网（仅指该业务VLAN需要覆盖到的网桥）

查看出现故障的业务VLAN，进行端到端的全部路径的查询。需要确认以下内容。

- 转发路径上每一个端口（网桥互连端口）上正确配置了相应的业务VLAN。

**命令：** `display interface interface-type interface-number`

```bash
[H3C]display interface GigabitEthernet1/0/1
GigabitEthernet1/0/1 current state: Up
PVID: 1
Mdit type: auto
Port link-type: trunk
VLAN passing: 1(default vlan), 100-110
VLAN permitted: 1(default vlan), 100-110
Trunk port encapsulation: IEEE 802.1q
```

- 转发路径上每一个端口（网桥互连端口）上正确学习到了相关MAC，且多次查看MAC是否频繁迁移。

**命令：** `display mac-address dynamic`

```bash
[H3C]display mac-address 0000-0000-0021
MAC ADDR          VLAN ID  STATE      PORT INDEX     TIME(s)
0000-0000-0021    100      LEARNED    GigabitEthernet1/0/1
--- 1 MAC address(es) found ---
```

如果存在频繁的业务MAC飘移，则很可能存在环路问题。需要进行环路的追踪。追踪的原则是判断业务MAC正常情况下应该从哪个端口收到，那么从其他的端口收到则可能是环路的端口。通常用display看到的端口就是发生环路的端口。

- 查看转发路径上每一个端口（网桥互连端口）上的流量统计情况，重点关注端口收发报文占用带宽的百分比，组播和广播流量大小。

**命令：** `display interface interface-type interface-number`


从端口统计数据看出，如果端口收发数据的百分比超过90%，并且广播和组播数据占据绝大部分比例时，很可能存在环路。需要追踪此类端口，来判断环路路径。判断方式是在环路路径上的端口收发的广播/组播数据量都很大，未在环路路径上的端口仅发方向的广播/组播流量大。
