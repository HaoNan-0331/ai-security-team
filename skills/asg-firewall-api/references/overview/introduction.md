# ASG 防火墙 API 概述

## 概述

ASG防火墙提供完整的REST API接口，支持通过编程方式对防火墙进行配置管理和运维操作。API采用Token认证方式，支持JSON格式的数据交换。

## 认证机制

### Token 认证

ASG防火墙使用Token认证机制，所有API请求都需要在URL参数中携带`api_key`参数。

**获取Token步骤：**

1. 登录ASG防火墙Web管理界面
2. 进入"系统管理" > "管理员" > "API管理"
3. 点击"生成Token"按钮
4. 复制生成的Token

**Token格式示例：** `56els9k7vt3ov6h7gaiuu04la5`

### 使用Token

**在URL中携带Token：**

```bash
# GET 请求示例
http://172.17.108.86/api/policy?api_key=YOUR_TOKEN&lang=cn

# POST 请求示例
curl -X POST http://172.17.108.86/api/policy?api_key=YOUR_TOKEN \
  -H "Content-Type: application/json" \
  -d '{"id":"1","protocol":"1","enable":"1"}'
```

## 请求格式

### 通用参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| api_key | String | 是 | 认证Token |
| lang | String | 否 | 语言（cn：中文，en：英文，默认cn） |
| page | Number | 否 | 当前页数（用于分页查询） |
| pageSize | Number | 否 | 每页条数（用于分页查询） |

### HTTP 方法

| 方法 | 说明 |
|------|------|
| GET | 查询数据 |
| POST | 新建数据 |
| PUT | 修改数据 |
| DELETE | 删除数据 |

### Content-Type

所有POST和PUT请求必须设置请求头：

```
Content-Type: application/json
```

## 响应格式

### 成功响应

```json
{
    "data": [
        {
            "id": "1",
            "name": "策略名称",
            "enable": "1"
        }
    ],
    "total": 1
}
```

### 错误响应

```json
{
    "code": "250",
    "str": "参数错误"
}
```

### 空响应

某些操作（如删除、修改）成功时可能返回空响应或无响应体。

## 连接信息

### 协议支持

- HTTP: `http://<IP地址>`
- HTTPS: `https://<IP地址>`（推荐）

### 默认端口

- HTTP: 80
- HTTPS: 443

### 示例连接信息

| 参数 | 示例 |
|------|------|
| IP地址 | 172.17.108.86 |
| 完整URL | https://172.17.108.86 |
| API端点 | /api/policy |
| 完整请求URL | https://172.17.108.86/api/policy?api_key=YOUR_TOKEN |

## 数据类型说明

### JSON 数据类型

ASG防火墙API使用JSON格式传输数据，支持以下数据类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| String | 字符串 | `"hello"`, `"192.168.1.1"` |
| Number | 数字 | `1`, `0`, `100` |
| Boolean | 布尔值（通常用Number表示） | `"1"`（true）, `"0"`（false） |
| Array | 数组 | `[{"item":1}, {"item":2}]` |
| Object | 对象 | `{"key": "value"}` |

### 特殊值约定

#### 启用/禁用状态

| 值 | 说明 |
|----|------|
| "0" | 禁用/关闭/不启用 |
| "1" | 启用/开启 |

#### 协议类型

| 值 | 说明 |
|----|------|
| "0" | IPv6 |
| "1" | IPv4 |

#### 策略动作

| 值 | 说明 |
|----|------|
| "1" | PERMIT（允许） |
| "2" | DENY（拒绝） |

#### 日志级别

| 值 | 说明 |
|----|------|
| "0" | 紧急 |
| "1" | 告警 |
| "2" | 严重 |
| "3" | 错误 |
| "4" | 警示 |
| "5" | 通知 |
| "6" | 信息 |

## 分页查询

### 分页参数

大多数GET查询接口支持分页：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| page | 当前页数 | 1 |
| pageSize | 每页条数 | 10（或更大） |

### 分页响应示例

```json
{
    "data": [
        {"id": "1", "name": "项目1"},
        {"id": "2", "name": "项目2"}
    ],
    "total": 100
}
```

- `data`: 当前页的数据数组
- `total`: 总记录数

## 批量操作

### 支持批量操作的接口

部分接口支持批量操作，通常通过数组参数传递：

```json
{
    "item": [
        {"name": "项1", "value": "值1"},
        {"name": "项2", "value": "值2"}
    ]
}
```

### 典型批量操作场景

- 批量添加地址对象
- 批量添加策略
- 批量删除规则

## 错误处理

### 错误码结构

```json
{
    "code": "错误码",
    "str": "错误描述信息"
}
```

### 常见错误处理建议

1. **认证失败**（Token无效）
   - 检查Token是否正确
   - 确认Token是否已过期
   - 重新生成Token

2. **参数错误**（code: 250）
   - 检查请求参数格式
   - 确认必填参数是否提供
   - 验证参数值范围

3. **对象不存在**（code: 87, 107）
   - 先执行查询操作确认对象存在
   - 检查对象名称或ID是否正确

4. **名称冲突**（code: 106）
   - 检查是否已存在同名对象
   - 使用不同的名称或先删除现有对象

## 安全建议

1. **Token安全**
   - Token具有较高权限，请妥善保管
   - 不要在代码中硬编码Token
   - 定期更换Token
   - 使用环境变量或配置文件存储Token

2. **HTTPS**
   - 生产环境建议使用HTTPS
   - 验证SSL证书（开发环境可临时禁用）

3. **日志记录**
   - 记录所有API操作
   - 不要在日志中记录敏感信息
   - 定期清理日志文件

## 最佳实践

1. **操作幂等性**
   - 添加操作前先查询是否已存在
   - 使用唯一的名称或ID
   - 处理名称冲突情况

2. **错误处理**
   - 检查每次操作的响应
   - 根据错误码进行相应处理
   - 记录错误日志便于排查

3. **性能优化**
   - 批量操作优于多次单次操作
   - 合理使用分页查询
   - 避免频繁轮询

4. **配置管理**
   - 使用配置文件管理连接信息
   - 分离开发和生产环境配置
   - 使用版本控制管理配置模板
